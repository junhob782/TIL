# [Study] 응용 프로그램 프로토콜과 데이터 전송 (1)

## 21. 응용 프로그램 프로토콜 (Application Protocol)
* **정의**: 하부 네트워크 망(TCP/IP)을 넘어, 최종 응용 프로그램 수준에서 주고받는 데이터의 **형식(Format)**, **의미(Semantics)**, 그리고 **처리 방식**을 정의한 규약.

---

## 22. 메시지 경계 구분 방법 (Message Boundary) 
> *TCP는 데이터의 경계가 없는 스트림(Stream) 방식이므로, 응용 계층에서 메시지의 끝을 식별하는 로직이 반드시 필요합니다.*

| 구분 | 송/수신 동작 방식 | 적합한 상황 (특징) |
| :--- | :--- | :--- |
| **1. 고정 길이** | - **송신자**: 항상 고정된 길이의 데이터를 보냄.<br>- **수신자**: 항상 정해진 고정 길이만큼만 데이터를 읽음. | 주고받을 **데이터의 길이 변동폭이 크지 않을 때** 적합함. |
| **2. 가변 길이 + EOR** | - **송신자**: 가변 길이 데이터 끝에 특별한 표시(`EOR`, End Of Record)를 붙여 보냄.<br>- **수신자**: `EOR` 패턴이 나올 때까지 계속 데이터를 읽음. | 생성될 **데이터의 길이를 미리 알 수 없을 때** 적합함. |
| **3. 크기(고정) + 데이터(가변)** | - **송신자**: 보낼 데이터의 크기(고정 길이)를 먼저 보내고, 이어서 실제 가변 데이터를 보냄.<br>- **수신자**: 크기를 먼저 읽어내고, 그 길이만큼 가변 데이터를 마저 읽음. | 생성 데이터의 길이를 미리 알고 있는 상황에서 **구현이 쉽고 처리 효율성이 가장 높음**. *(실무 표준)* |
| **4. 가변 길이 + 종료** | - **송신자**: 가변 길이 데이터 전송 후 즉시 연결을 정상 종료함.<br>- **수신자**: `recv()` 함수의 리턴값이 `0`(정상 종료, EOF)이 될 때까지 계속 읽음. | 한쪽에서 **일방적으로 가변 길이 데이터를 쭉 보낼 때** (예: 파일 다운로드) 적합함. |

## 23. 다양한 데이터 전송 방식 상세 구현

### 23.1 고정 길이 데이터 전송 (Fixed-Length)
* **구현 방식**: 서버와 클라이언트 모두 사전에 약속된 **크기가 같은 버퍼**를 정의해두고 데이터를 주고받음.
* **데이터 타입 참고**: `wchar_t` (Wide Character)
    * 유니코드(Unicode) 등 다국어 문자를 표현하기 위해 2바이트 이상을 사용하는 확장 문자형. 고정 길이 버퍼 설계 시 바이트 계산에 주의해야 함.

### 23.2 가변 길이 데이터 전송 (Variable-Length with EOR)
* **구현 방식**: 데이터의 끝(경계)을 구분하기 위해 `\r\n` 이나 특수 기호 등 **EOR로 사용할 데이터 패턴**을 사전에 명확히 정해야 함. (예: HTTP 헤더의 끝을 알리는 빈 줄).

### 23.3 고정 길이 + 가변 길이 데이터 전송 (Header + Payload)
> *아키텍트 추천: 가장 버그가 적고 깔끔한 데이터 전송 설계 방식입니다.*

* **조건**: 송신 측에서 전송하기 전에 가변 길이 데이터의 총 크기를 미리 계산할 수 있어야 효과적임.
* **수신 측 2-Step 읽기 작업**:
    1. **고정 길이 데이터 수신**: 먼저 4바이트(int) 등의 고정 헤더를 읽어 뒤따라올 데이터의 `Size`를 파악함.
    2. **가변 길이 데이터 수신**: 파악한 `Size`만큼 두 번째 읽기 작업을 수행하여 가변 길이 데이터의 경계를 완벽하게 구분해 냄.
