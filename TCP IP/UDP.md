# [Study] UDP 프로토콜과 다자간 통신 (Broadcasting)

## 39. UDP (User Datagram Protocol) 개요
* **TCP와 UDP의 공통점**:
    1. **포트(Port) 번호**를 이용하여 통신의 종착점(응용 프로그램 주소)을 지정함.
    2. 헤더에 포함된 체크섬(Checksum)을 통해 **데이터 오류를 체크**함.

### 39.1 UDP의 4가지 핵심 특징
1. **비연결성**: 연결 설정(3-Way Handshake)을 하지 않으므로 TCP의 `connect()` 함수 호출이 불필요함.
2. **신뢰성 미보장**: 프로토콜 수준에서 신뢰성 있는 데이터 전송을 보장하지 않음 (패킷 유실 시 재전송 안 함). 
   * *해결책*: 필요하다면 응용 프로그램 수준(Application Layer)에서 확인 응답(ACK) 등의 신뢰성 전송 기능을 직접 구현해야 함.
3. **다자간 통신 용이**: 간단한 소켓 함수 호출 절차만 따르면 1:N, N:M 등 다자간 통신을 매우 쉽게 구현할 수 있음.
4. **데이터 경계 유지**: TCP(스트림 방식)와 달리 패킷(데이터그램) 자체가 독립적인 경계를 가지므로, 응용 프로그램이 데이터 경계 구분을 위한 작업을 별도로 할 필요가 없음.

## 40. TCP vs UDP 비교 총정리 

| 구분 | TCP (Transmission Control Protocol) | UDP (User Datagram Protocol) |
| :--- | :--- | :--- |
| **연결 방식** | **연결형 프로토콜** (연결 설정 후 통신 가능) | **비연결형 프로토콜** (연결 설정 없이 즉시 통신 가능) |
| **신뢰성** | **신뢰성 보장** (필요시 데이터 자동 재전송) | **신뢰성 없음** (데이터 재전송하지 않음) |
| **통신 형태** | 1:1 통신 전용 (Unicast) | 1:1 통신 및 1:多, 多:多 통신 모두 가능 |
| **데이터 서비스** | **데이터 경계 구분 안 함** (바이트 스트림 서비스) | **데이터 경계를 명확히 구분함** (데이터그램 서비스) |

## 41. UDP 데이터 전송 함수
* **`sendto()`**: 목적지 주소(IP/Port)를 지정하여 응용 프로그램의 데이터를 운영체제의 '송신 버퍼'에 복사함으로써 데이터를 전송함.
* **`recvfrom()`**: 운영체제의 '수신 버퍼'에 도착한 데이터를 응용 프로그램 버퍼에 복사함 (송신자의 주소 정보도 함께 받아옴).

## 42. UDP 서버/클라이언트의 IPv6 마이그레이션
> *기존 IPv4 기반 UDP 코드를 IPv6로 전환하기 위한 필수 변경 사항*

1. **새로운 헤더 파일 추가**: 윈도우 환경에 한해 `ws2tcpip.h` 헤더를 추가로 포함해야 함.
2. **소켓 생성 체계 변경**: `socket()` 생성 시 `AF_INET` 대신 **`AF_INET6`**를 사용함.
3. **소켓 주소 구조체 변경**: 16바이트 크기의 `sockaddr_in` 대신, 128비트 주소를 수용할 수 있는 28바이트 크기의 **`sockaddr_in6`** 구조체를 사용함.

## 43. 네트워크 캐스팅 (통신 개체 간 상호 작용 방식) 
| 캐스팅 종류 | 통신 대상 | 개념 요약 |
| :--- | :--- | :--- |
| **유니캐스팅 (Unicasting)** | **1:1** | 특정 목적지 하나를 지정하여 통신하는 방식. |
| **브로드캐스팅 (Broadcasting)** | **1:전체** | 상대방 전체(동일 네트워크 내)에게 통신하는 방식. |
| **멀티캐스팅 (Multicasting)** | **1:특정 그룹** | 내가 지정한 특정 다수(Multicast Group)에게만 통신하는 방식. |
| **애니캐스팅 (Anycasting)** | **1:가장 가까운 1** | 점 구조 (가장 가까운 노드 하나로 라우팅됨). |

## 44. 브로드캐스팅 (Broadcasting) 심화
* **개념**: 송신자가 보낸 데이터 하나를 다수의 수신자가 받는 방식. 송신자 관점에서는 데이터 복사본을 직접 여러 개 만들어 보내는 것이 아니므로(라우터/스위치가 복사함) 네트워크 대역폭 측면에서 상당히 효율적인 기술임.

### 44.1 브로드캐스트 데이터 전송 절차
1. **브로드캐스팅 활성화**: 소켓의 옵션을 변경하여 브로드캐스트 전송을 허용해야 함. *(아키텍트 노트: C언어에서는 `setsockopt()` 함수를 호출하여 `SO_BROADCAST` 옵션을 활성화합니다.)*
2. **목적지 설정 및 전송**: 목적지 IP를 브로드캐스트 주소로 설정하여 `sendto()` 데이터 전송.

### 44.2 브로드캐스트 주소의 종류
1. **네트워크 브로드캐스트 (Local Broadcast)**: 현재 내가 속한 동네(로컬 물리 네트워크)의 모든 호스트에게 묻지마 전송. (주소 예: `255.255.255.255`)
2. **서브넷 브로드캐스트 (Subnet-directed)**: 특정 원격 네트워크(다른 동네) 단위에 있는 '모든' 호스트에게 전송.
3. **지역 브로드캐스트 (All-subnets-directed)**: 원격 네트워크를 더 잘게 쪼갠 '특정 서브넷(구역)' 내의 모든 호스트에게 전송.
# [Study] 소켓 옵션(Socket Options)과 멀티캐스팅 제어

## 45. 소켓 프로그래밍 모델과 옵션 개요
* **계층 모델**: `응용 프로그램` $\rightarrow$ `소켓 코드` $\rightarrow$ `프로토콜 구현 코드 (TCP/IP)`
* **소켓 옵션의 목적**: 소켓 함수의 기본 동작을 변경하여, 소켓 코드와 하부 프로토콜 구현 코드를 개발자가 세부적으로 제어하기 위함.
* **처리 주체에 따른 분류**:
    1. **소켓 코드 처리 옵션**: 옵션 설정 시 소켓 계층(API 레벨)에서 해석 후 처리됨.
    2. **프로토콜 구현 코드 처리 옵션**: 옵션 설정 시 하부 네트워크 프로토콜 계층으로 내려가 해석 후 처리됨.

## 46. 주요 소켓 옵션 총정리 

| 옵션명 | 용도 및 특징 | 비고 |
| :--- | :--- | :--- |
| **`SO_REUSEADDR`** | 지역 주소(IP, 포트 번호) 재사용 여부 설정 (**매우 중요**) | 서버 재시작 시 `bind` 에러 방지 |
| **`SO_BROADCAST`** | 브로드캐스트 데이터 전송 기능 활성화 | **UDP 전용** (TCP 사용 불가) |
| **`SO_KEEPALIVE`** | TCP 프로토콜 수준에서 상대방의 생존 여부 확인 (주기적 패킷 전송) | TCP 전용 |
| **`SO_LINGER`** | `close()`/`closesocket()` 호출 시 송신 버퍼에 남은 데이터 처리 방식 변경 | 우아한 종료와 연관 |
| **`SO_SNDBUF` / `SO_RCVBUF`** | 소켓의 송신 및 수신 버퍼 크기 변경 | |
| **`SO_SNDTIMEO` / `SO_RCVTIMEO`** | 데이터 전송 함수(`send`/`recv`)의 타임아웃(Timeout) 설정 | 무한 대기(Block) 방지 |

### 46.1 SO_REUSEADDR 상세 분석 (아키텍트의 강조)
* **목적**: 현재 사용 중인 IP 주소와 포트 번호를 재사용하여 `bind()` 함수를 즉시 다시 호출할 수 있게 함.
* **적용 상황**:
    1. TCP 서버가 비정상 종료되거나 재실행될 때, `TIME_WAIT` 상태로 인해 **`bind()` 함수에서 오류(Address already in use)가 발생하는 것을 방지**함.
    2. 여러 IP 주소를 보유한 다중 홈(Multi-homed) 호스트에서 같은 기능의 서버를 IP 주소별로 따로 운용할 수 있게 함.
    3. 멀티캐스팅 응용 프로그램이 같은 포트 번호를 공유하여 사용할 수 있게 함.

### 46.2 SO_KEEPALIVE 상세 동작 원리
* **정상 응답**: 상대 TCP가 정해진 시간 안에 응답(ACK)하는 경우 $\rightarrow$ 응용 프로그램 정상 작동 유지.
* **무응답 (단절)**: 상대 TCP가 정해진 시간 안에 응답하지 않는 경우 $\rightarrow$ 자동으로 소켓을 닫아 시스템 자원 소모(좀비 커넥션)를 막음.
* **RST 패킷 응답**: 상대 컴퓨터가 비정상 재부팅되는 등 RST(Reset) 패킷으로 응답하는 경우 $\rightarrow$ 자동으로 소켓을 닫음.

### 46.3 아키텍트의 실무 팁 (Graceful Shutdown)
> "소켓을 닫을 때 단순히 `close()`나 `SO_LINGER` 옵션에만 의존하기보다는, **`shutdown()` 함수를 활용하여 송신/수신 스트림을 명시적으로 절반씩 닫는(Half-close) 우아한 종료 절차**를 거치는 것이 실무 서버 설계의 표준입니다."

## 47. 멀티캐스팅 (Multicasting) 심화 옵션
* **멀티캐스팅의 3대 특징**:
    1. 그룹 가입과 탈퇴가 자유롭고, 그룹 구성원 모두가 평등한 권한을 가짐.
    2. 멀티캐스트 데이터를 **받으려면(수신)** 반드시 해당 멀티캐스트 그룹에 **가입(Join)**해야 함.
    3. 멀티캐스트 데이터를 **보내려면(송신)** 해당 그룹에 굳이 **가입할 필요 없음**.

### 47.1 멀티캐스트 전용 소켓 옵션
| 옵션명 (IPv4 / IPv6) | 기능 설명 |
| :--- | :--- |
| `IP_MULTICAST_IF` / `IPV6_MULTICAST_IF` | IP 주소를 두 개 이상 보유한 호스트에서, 멀티캐스트 데이터를 내보낼 **네트워크 인터페이스(랜카드)를 명시적으로 선택**함. |
| `IP_MULTICAST_TTL` / `IPV6_MULTICAST_HOPS` | 데이터가 라우터를 통과할 수 있는 한계치인 **TTL(Time To Live) / Hop 값을 변경**하여 전송 범위를 제한함. |
| `IP_MULTICAST_LOOP` / `IPV6_MULTICAST_LOOP` | 그룹에 가입한 응용 프로그램이 데이터를 보낼 때, **자신도 루프백(Loopback)으로 다시 받을지를 결정**함. |
| `IP_ADD_MEMBERSHIP` / **`IPV6_JOIN_GROUP`** | 멀티캐스트 그룹에 **가입(Join)**하는 코드. |
| `IP_DROP_MEMBERSHIP` / **`IPV6_LEAVE_GROUP`** | 멀티캐스트 그룹에서 **탈퇴(Leave)**하는 코드. |
*(※ 아키텍트 노트: IPv6의 가입/탈퇴 표준 매크로명은 ADD/DROP 대신 JOIN/LEAVE를 사용합니다.)*

## 48. TCP_NODELAY 옵션과 Nagle 알고리즘 
* **`TCP_NODELAY` 옵션**: TCP 통신 시 기본적으로 켜져 있는 **네이글(Nagle) 알고리즘의 작동 여부(On/Off)**를 결정함. (옵션을 활성화하면 Nagle 알고리즘을 끔).

### 48.1 Nagle 알고리즘의 동작 방식
* **조건 1**: 보낼 데이터가 `MSS`(Maximum Segment Size, 최대 세그먼트 크기)로 정의된 크기만큼 쌓이면 상대편에 무조건 즉시 보냄.
* **조건 2**: 보낼 데이터가 `MSS`보다 작으면, 이전에 보낸 데이터에 대한 **`ACK` 응답이 올 때까지 기다림**. `ACK`가 도착하면 그제서야 버퍼에 모인 작은 데이터를 한 번에 포장해서 상대편에 보냄.

### 48.2 Nagle 알고리즘의 장단점
* **장점**: 작은 크기의 패킷이 불필요하게 많이 생성되어 헤더 오버헤드가 커지는 일을 방지함 $\rightarrow$ **전체 네트워크 트래픽(혼잡) 감소**.
* **단점**: 데이터가 충분히 쌓일 때까지, 또는 이전 데이터의 `ACK`가 도달할 때까지 송신이 지연(대기)되므로 $\rightarrow$ **응용 프로그램의 반응 시간(Latency)이 길어질 수 있음**.
    * *(적용 사례: FPS 게임이나 실시간 주식 트레이딩 등 빠른 반응이 생명인 서버에서는 반드시 `TCP_NODELAY` 옵션을 사용해 Nagle 알고리즘을 끕니다.)*
