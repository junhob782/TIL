# [Study] Windows GUI 프로그래밍과 네트워크 스레드 분리

## 49. Windows GUI와 API 기초
* **GUI (Graphical User Interface)**: 다양한 API로 구현된 편리하고 화려한 시각적 사용자 인터페이스를 제공하며, 기본적으로 **메시지 구동(Message-Driven) 구조**로 동작함.
* **API (Application Programming Interface)**: 윈도우 운영체제(OS)가 응용 프로그램에 제공하는 C언어 기반의 시스템 함수 집합 (Win32 API).
* **메시지 (Message)**: 윈도우 운영체제가 응용 프로그램의 외부(마우스 클릭, 키보드 입력 등)나 내부(타이머 등)에 변화가 생겼음을 해당 프로그램에 알리는 데 사용하는 통신 수단.

---

## 50. 메시지 구동 동작 원리 
> *윈도우 응용 프로그램이 이벤트를 처리하는 3단계 핵심 흐름*

1. **이벤트 발생 및 시스템 큐 저장**: 마우스 클릭 등의 이벤트가 발생하면, 운영체제가 이를 감지하여 '시스템 메시지 큐'에 우선 저장함.
2. **응용 프로그램 큐로 전달**: 운영체제는 시스템 큐에 저장된 메시지를 분석하여, 타겟이 되는 적절한 '응용 프로그램의 메시지 큐(Message Queue)'로 보냄.
3. **메시지 루프 (Message Loop)**: 응용 프로그램은 `while` 문을 돌며 자신의 메시지 큐를 끊임없이 감시함. 메시지가 들어오면 하나씩 꺼내어 처리하고, 처리할 메시지가 없을 때는 CPU를 낭비하지 않고 대기(Idle) 상태에 들어감.

### 50.1 메시지 처리자 (Message Handler & Window Procedure)
* **메시지 핸들러**: 특정 메시지(예: 마우스 클릭)를 받았을 때 프로그램이 어떻게 동작할지 결정하는 실제 코드 블록.
* **윈도우 프로시저 (Window Procedure / `WndProc`)**: 이러한 메시지 핸들러들이 모여 있는 거대한 `switch-case` 형태의 함수(집합).

## 51. WinMain 함수 (GUI 프로그램의 진입점)
* 콘솔 프로그램의 진입점이 `main()`이라면, Windows GUI 응용 프로그램의 실행 시작점은 **`WinMain()`** 함수임.

| 주요 인수(Parameter) | 설명 및 용도 |
| :--- | :--- |
| **`hInstance`** | **인스턴스 핸들**. 현재 실행 중인 프로그램 자체를 식별하는 값으로, 실행 파일(.exe)에 포함된 아이콘, 이미지 등 각종 리소스에 접근할 때 필수적으로 사용함. |
| **`lpCmdLine`** | **명령행 인수 문자열**. 프로그램 실행 시 뒤에 덧붙여 전달된 텍스트 파라미터 (C언어의 `argv`와 유사함). |
| **`nCmdShow`** | **창 표시 상태**. 프로그램 시작 시 메인 윈도우를 어떻게 보여줄지(최대화, 최소화, 기본 크기 등)를 제어하는 정수값. |

## 52. GUI 컨트롤과 문자열 (TCHAR)
* **컨트롤 (Control)**: 버튼(Button), 텍스트 박스(Edit), 리스트(Listbox) 등 표준화된 형태와 특성을 제공하여 사용자의 입력을 받거나 출력하는 일종의 작은 자식 윈도우.
* **`TCHAR` (와이드 캐릭터 지원)**: 한글과 같은 다국어를 깨짐 없이 2바이트(Unicode)로 표현하기 위해, 윈도우 API에서 권장하는 가변형 문자열 매크로 데이터 타입 (`wchar_t` 기반).

## 53. [핵심] GUI 윈도우의 메시지 처리 지연 현상과 해결책
* **문제 상황 (UI Freeze)**: `accept()`나 `recv()` 같은 소켓 함수는 데이터가 올 때까지 무한정 대기(Block)하는 특성이 있음. 만약 이 함수들을 GUI를 그리는 메인 스레드(메시지 루프)에서 호출하면, **메시지 처리가 지연되어 프로그램 창이 하얗게 멈추는 '응답 없음' 상태**가 됨.
* **해결 방법**: 
    1. 무한 대기가 발생하는 **소켓 통신 코드를 별도의 워커 스레드(Worker Thread)로 분리**하여 백그라운드에서 돌려야 함.
    2. 메인 스레드(UI)와 통신 스레드(소켓)가 같은 변수나 텍스트 박스(공유 데이터)에 접근할 경우, 충돌 방지를 위해 반드시 **스레드 동기화 기법(Mutex, Critical Section 등)을 사용**하여 보호해야 함.

## 54. 대화상자 (Dialog Box) 기반 프로그래밍
* **개념**: 다양한 컨트롤(버튼, 입력창 등)을 직관적으로 포함하여 사용자와 상호작용하는 특수한 형태의 윈도우.

### 54.1 대화상자 디자인부터 화면에 보이기까지의 4단계 절차
1. **리소스 추가**: Visual Studio 프로젝트에 리소스 스크립트 파일(`*.rc`)을 추가함.
2. **시각적 디자인**: 리소스 파일에 대화상자를 추가하고, Visual Studio의 '리소스 편집기'를 이용해 버튼 등을 드래그 앤 드롭으로 시각적으로 배치(디자인)함.
3. **리소스 컴파일**: 프로젝트를 빌드(Build)하면, C/C++ 컴파일과는 별개로 '리소스 컴파일러(RC.exe)'가 실행되어 텍스트 형태인 `*.rc` 파일을 이진(Binary) 형식인 `*.res` 객체 파일로 변환함.
4. **런타임 생성**: 프로그램 실행 중, 대화상자 생성을 요청하는 Win32 API 함수(`CreateDialog`, `DialogBox` 등)를 호출함. 운영체제는 실행 파일(.exe) 내부에 결합된 `*.res` 리소스 데이터를 토대로 화면에 실제 대화상자를 짠 하고 띄움.
